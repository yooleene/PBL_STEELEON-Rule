# 규정 데이터 RAG 적용을 위한 상세 수정 가이드 (초보자 설명 포함)

---

## 1. 문서 목적
본 문서는 규정을 RAG 시스템의 소스 문서로 활용할 때 발생할 수 있는 위험을 줄이기 위해 작성됨.

### 설명
- **RAG란?**: Retrieval-Augmented Generation의 약자로, 문서를 검색해서 답변을 생성하는 AI 시스템임.
- 규정 문서는 사람이 읽기에는 괜찮지만, AI가 이해하기에는 구조가 불명확한 경우가 많음.
- 따라서 AI가 잘 이해하도록 문서를 "정리(정제 + 정규화)"하는 것이 목적임.

---

## 2. 핵심 결론
- 원문 보관용: 적합
- RAG용 canonical source: 부적합

### 설명
- **원문 보관용**: 감사나 법적 근거를 위해 원본은 반드시 유지해야 함.
- **RAG용**: AI가 검색하고 답변하기 위한 문서는 따로 만들어야 함.

### 주요 문제 설명
- 개정이력/본문 혼재 → 과거 내용과 현재 내용이 섞여 있음
- 조문번호 불안정 → 번호 체계가 일정하지 않음
- 표 구조 손상 → 표가 깨져서 AI가 이해하기 어려움
- 삭제 조항 포함 → 이미 삭제된 규정이 남아 있음
- 참조 문서 의존성 → 다른 문서를 봐야 이해 가능
- 숫자 비구조화 → 금액, 기준 등이 텍스트로 흩어져 있음

---

## 3. 우선순위별 수정 사항

### 설명
- **1순위**: AI가 답변을 못하는 문제를 바로 해결
- **2순위**: 정확도 향상
- **3순위**: 운영 효율 개선

---

## 4. 상세 수정 가이드

### 4.1 개정이력 분리
- 개정이력 별도 문서화
- 현행본 분리

#### 설명
- 개정이력은 과거 변경 기록임
- AI는 최신 규정만 기준으로 답해야 하므로 분리 필요

---

### 4.2 삭제 조항 제거
- deleted 상태 관리
- 검색 제외

#### 설명
- 삭제된 규정은 사용자에게 혼란을 줌
- DB에서는 보관하지만 검색에서는 제외해야 함

---

### 4.3 조문 번호 정리
- 내부 ID 생성

#### 설명
- 조문 번호가 바뀌어도 동일 조항을 추적하기 위해 고유 ID 필요

---

### 4.4 표 구조화
- JSON/CSV 변환
- 셀 단위 검색

#### 설명
- 표를 그대로 두면 AI가 이해 못함
- 구조화하면 "조건-결과" 형태로 검색 가능

---

### 4.5 날짜 정리
- 시행일/개정일 분리

#### 설명
- 날짜가 섞이면 최신 기준 판단이 어려움

---

### 4.6 참조 문서 연결
- multi-hop retrieval

#### 설명
- 한 문서만으로 답이 안 되는 경우 연결 필요

---

### 4.7 숫자 구조화
- facts 테이블 구축

#### 설명
- 금액, 기간 등은 별도로 관리해야 정확한 답변 가능

---

### 4.8 정규화
- 날짜/용어 통일
- 동의어 사전

#### 설명
- 같은 의미인데 다른 표현이면 AI가 다른 것으로 인식함

---

## 5. 전처리 순서

### 설명
순서를 지켜야 데이터 품질이 유지됨

1. 원문 보존 → 원본 유지
2. 현행본 분리 → 최신만 추출
3. 조문 파싱 → 구조 분해
4. 표 구조화 → 데이터화
5. 정규화 → 표현 통일
6. 메타데이터 → 검색 정보 추가
7. 인덱스 분리 → 검색 최적화
8. QA → 검증

---

## 6. 메타데이터 설계

### 설명
- 메타데이터는 "검색을 위한 정보"임

| 항목 | 설명 |
|------|------|
| document_name | 문서 이름 |
| effective_date | 시행일 |
| clause_id | 조항 고유 ID |
| status | 활성/삭제 상태 |
